# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Unity 6 (6000.3.9f1) C# mobile application for WhatsApp and Telegram bot automation. Manages chat interfaces, bot creation, and workflow automation with products, services, and messaging features. Targets iOS and Android via URP (Universal Render Pipeline).

## Build & Development

- **Unity version**: 6000.3.9f1 — open through Unity Hub
- **Rendering**: Universal Render Pipeline (URP 17.3.0)
- **Solution**: `Automation.sln` (auto-generated by Unity, gitignored)
- **NuGet packages**: managed via NuGetForUnity in "In Packages folder" placement — everything lives under `Packages/nuget-packages/` (`NuGet.config`, `packages.config`, `InstalledPackages/`). Keep NuGet files out of `Assets/`; the plugin reads `Packages/nuget-packages/NuGet.config` first and only falls back to `Assets/NuGet.config`
- **Git LFS**: enabled for binary assets (textures, audio, fonts, plugins) — see `.gitattributes`

Command-line build example:
```
Unity -batchmode -nographics -projectPath . -buildTarget Android -quit
```

### Running EditMode tests

EditMode tests live in `Assets/Tests/Editor/Chat/` (no asmdef — they compile into the predefined `Assembly-CSharp-Editor`). Two ways to run them:

- **Editor closed (headless, hands-off)** — `Tools/run-tests-headless.sh`. Launches Unity in batch mode (`-runTests -batchmode -nographics -testPlatform EditMode`, never `-quit`), parses the NUnit3 result, and writes outputs to `Tools/test-output/` (gitignored — must NOT be under `Temp/`, which Unity wipes on launch). Refuses if the Editor has the project open (the project lock is single-instance). Pass a `-testFilter` regex as `$1` to scope the run.
- **Editor open** — `Assets/Editor/ClaudeTestBridge.cs`: drop `Temp/claude/run-tests.trigger`, read `Temp/claude/test-summary.json`. Runs the suite inside the already-open Editor (which must be focused).

## Architecture

**Single scene**: `Assets/Scenes/Main.unity` — all UI is canvas-based, toggled via `GameObject.SetActive()`. No scene transitions.

### Navigation

Bottom tab index 2 (formerly «New») now points to `Screen_Dashboard` (the «Сводка» conversation-outcomes dashboard, `DashboardPage`). The bot-creation form (`Screen_New`) is no longer a tab — it's a slide-in overlay (`AddBotPanel`, `AddBotPanel.Instance`) opened from the Bots page (header `+`, empty-state CTA, or auto-opened on zero bots). `NavRestructureBuilder.ReorderScreens` enforces a deterministic `ScreenContainer` child order: all tab screens first (`Screen_Dashboard` after `Screen_Profile`), then `Screen_New` (the overlay), then `WhatsappAuth`/`TelegramAuth` LAST so auth pages always render above the opaque overlay form — preserve this order when adding new screens.

### Chat data flow

```
wappi.pro API
  └── ChatManager.SyncAllChats()         → ChatsResponse (JsonUtility) → ChatViewModel list
  └── ChatManager.GetMessagesRoutine()   → MessagesResponseRaw (JsonConvert) → RawMessage[]
        └── Normalize(RawMessage)        → NormalizedMessage
              └── CreateViewModel()      → MessageViewModel
                    └── OnBatchMessagesLoaded event → MessageListView renders prefabs
```

### Caching strategy

- Chat list: `persistentDataPath/BotCache/{botId}/chats.json` (per-bot; the root-level `all_chats_cache.json` is legacy with no code references)
- Message history: `ChatHistoryCache` (per `chatId`, max 100 messages)
- Media/thumbnails: `MediaCacheManager` saves to disk, keyed by URL
- Background sync pattern: on chat open, load from cache instantly, then quietly diff against server response.

**Key third-party plugins** (in `Assets/Plugins/`): DOTween (animation), NativeFilePicker, NativeGallery, unity.webp. Additional UPM packages: NativeShare, RoundedCorners.

## External APIs

The app communicates with four external services via `UnityWebRequest` + coroutines (`IEnumerator` + `yield return SendWebRequest()`). Auth via headers or URL path.

### Wappi.pro (WhatsApp)
- **Base**: `https://wappi.pro/api/sync/` (some endpoints under `https://wappi.pro/api/` directly, e.g. `profile/add`, `profile/delete`, `profile/logout`)
- **Auth**: `Authorization` header with `Manager.wappiAuthToken`
- **Endpoints**:
  - `chats/filter` — filtered chat list
  - `messages/get` — paginated message history (requires `chat_id`)
  - `messages/all/get` — all messages (bulk)
  - `messages/id/get` — fetch a SINGLE message by id (`profile_id` + `message_id`, no `chat_id`); returns `{ status:"done", message:{ id/type/body/caption/senderName/contact_name/fromMe/… } }`. Used to recover a quoted reply's real text when `reply_message` is missing/echoed (see Incoming replies)
  - `message/media/download` — download media by message ID
  - `message/send` — send a message; add `quoted_message_id` (omit when null) to send a **reply** quoting another message
  - `message/reaction` — send/remove an emoji reaction (body `{ body, message_id }`; empty `body` removes; targets by `message_id`, no recipient)
  - `chat/delete` — delete a chat (POST, body `{ recipient }` where recipient = chatId minus `@c.us`; success `status:"done"`). A successful call deletes the chat in WhatsApp; Wappi then reports it back in `chats/filter` with a sticky `isDeleted:true`. Backs swipe-to-delete: `ChatManager.DeleteChat.cs` removes the row optimistically (rollback on failure) + evicts `ChatHistoryCache`; `ParseChatsJson` hides any chat with `isDeleted == true`, so a deletion sticks across sync/bot-switch/restart. **Confirmed platform limits** (don't try to "fix" these — they're Wappi/WhatsApp constraints): the call is per-Wappi-profile and does NOT propagate to other bots; deletes made directly in the WhatsApp app do NOT set `isDeleted` and are invisible to us; re-deleting an already-`isDeleted` chat returns HTTP 400 `deleteChat error`. (A persistent per-bot watermark + revival design was built and then reverted — `isDeleted` is sticky-true even after new activity, so there's no reliable revival signal and the extra layer wasn't worth it.)
  - **Incoming replies**: a reply message carries `isReply: true` + (often, but not always) a `reply_message` snapshot of the quoted original — keys `id` / `type` / `body` / `caption` / `contact_name` (the quoted sender; NOT `senderName`) / `file_name`; no `fromMe`. Resolved by `ReplyParser` (cache-by-id → snapshot → placeholder) into flat `quoted*` fields carried through the 4-layer message pipeline; rendered as a quoted card by `MessageItemView`. The `reply_message` id is always correct, but its `body` is sometimes absent or **echoes the replying message's own text** — `ReplyParser.FromSnapshot` detects that (snapshot body == own raw body) and blanks it, then the card is recovered by fetching the original via `messages/id/get` (`QuotedMessageCache` + `ChatManager.QuoteResolve.cs`, mirroring the reaction-target resolver). Reply triggers: swipe-right (`SwipeToReply`) + the long-press action menu (`ReactionBarController`).
  - `qr/get` — QR code for WhatsApp login
  - `auth/code` — request phone pairing code (`profile_id` + `phone`; response `code` is displayed in-app, user types it into WhatsApp). **Confirmed platform limit**: WhatsApp refuses a repeat code for the same profile for ~2 minutes. The resend button therefore silently deletes + recreates the Wappi profile (same name) behind the loading overlay and requests the code on the fresh profile — `Manager.GetWhatsappCode` → `RecreateWhatsappProfileForNewCode` (with a pre-delete `get/status` guard so a just-authorized profile is never destroyed). Don't "simplify" resend back to a plain re-request; safe in both wizard and settings flows because no n8n workflow exists while the code panel is reachable (workflows are created after auth; change-number/logout delete them first)
  - `get/status` — connection status
  - `contact/info` — contact details
  - `profile/add`, `profile/delete`, `profile/logout`

### Wappi.pro (Telegram)
- **Base**: `https://wappi.pro/tapi/sync/` (profile management under `https://wappi.pro/tapi/`)
- **Auth**: same `Authorization` header
- **Endpoints**: `auth/qr`, `auth/phone` (POST), `auth/code` (POST), `get/status`, `profile/add`, `profile/delete`, `profile/logout`

### n8n (Workflow Automation)
- **Base**: `https://bagkz.app.n8n.cloud/`
- **Auth**: `X-N8N-API-KEY` header with `Manager.n8nAPIKey`
- **API endpoints** (`/api/v1/workflows/`):
  - `{id}/activate`, `{id}/deactivate` — POST with an explicit `Content-Type: application/json` header (empty body is fine). Unity's libcurl transport stamps `application/x-www-form-urlencoded` onto a POST with no upload handler and n8n's REST API 415s any non-JSON content type, so "bodyless" alone is not enough. The Enable coroutines also skip the `""`/`"-1"` sentinel ids (channel never authed → no workflow → guaranteed 404)
  - `DELETE {id}`
- **Webhook endpoints**:
  - `/webhook/CreateWhatsappWorkflow`, `/webhook/CreateTelegramWorkflow`
  - `/webhook/EditWhatsappWorkflow`, `/webhook/EditTelegramWorkflow` — all 4 forms now include `BusinessTypeId` (stable kebab-case id from `BusinessTypes.asset`, one of the 6 KZ Tier-1 verticals: `auto_parts`/`wholesale`/`flowers`/`kaspi_seller`/`education`/`phone_repair`); a `Vertical Prompt` Code node in each workflow injects the matching vertical main prompt (canonical sources `Tools/n8n/prompts/<id>.md`, composed with `_core.md`, injected by `Tools/n8n/inject-prompts.py`) into the created/edited bot workflow's systemMessage head — empty/unknown id falls back to the legacy generic template prompt (Create) or preserve-head (Edit). Bots saved with a pre-vertical legacy id (`car_service`, `dentist`, …) show a «Тип не выбран» placeholder in the settings dropdown and deliberately send `BusinessTypeId=""` (keeping the stored id in PlayerPrefs) so a settings save never silently migrates their prompt head — see `Manager.ApplyBusinessTypeToDropdown`. The Промпт field in BotSettings remains Additional Instructions only
  - `/webhook/UploadFile` — price-list upload. The app mints a GUID `fileId` per file (form field) and persists `{fileId, name, size, date}` per bot in PlayerPrefs (`UploadedFilesStore`); the workflow stamps `fileId` onto every RAG chunk's metadata in Supabase `documents`. For documents, the app converts structured formats (xml/csv/tsv/xls/xlsx/xlsm/docx/rtf/html — with cp1251/UTF-16 encoding sniffing, ru-locale ';' CSV, 1C "fake .xls" HTML/SpreadsheetML detection) to plain text CLIENT-side (see `Converters/`) — deliberate: the converters emit RAG-ready, entity-labeled text that generic server extraction can't, and n8n's Extract From File can't parse docx (and its rtf op only reads tables). For photos (jpg/jpeg/png/webp/heic client-side sources), the app uses `ImageUploadPreprocessor` to decode, downscale to 2048px max, and re-encode as JPEG q85 before sending. The workflow ingests text/plain + PDF (extension routing case-insensitive) via document converters, and routes JPEG via a Switch branch to OpenAI gpt-4o-mini vision extraction (emits same entity-labeled `{contentType}[N]:...` lines); unreadable photos get HTTP 422 `{"success": false, "error": "no_price_data"}` (no retry). All uploads (documents and photos) archive the original bytes to the private Supabase Storage bucket `price-lists` keyed by `fileId`, so re-indexing (re-chunk/re-embed) never requires users to re-upload
  - `/webhook/DeleteFile` — body `{ fileId }`; deletes that file's RAG chunks (`metadata->>'fileId'`) and its stored original from the `price-lists` bucket, returns `{ success, deletedChunks }`. Backs the per-file ✕ in the BotSettings "Прайс-листы" list (`BotSettings.Files.cs`); files uploaded before fileId stamping can't be individually deleted
  - `/webhook/DeleteBotFiles` — body `{ botWaId, botTgId }`; sweeps ALL of a deleted bot's RAG chunks (matched by either workflow id, guarding the `"-1"` unauthed sentinel so shared-unauthed chunks are never touched) plus each file's stored original (including legacy chunks with no fileId — those just have no bucket object). Returns `{ success, deletedChunks, deletedFiles }`. Called fire-and-forget by `Bot.DeleteBot` via `Manager.DeleteBotFilesOnServer` (the Bot GameObject destroys itself, so the coroutine lives on Manager)
  - `/webhook/DashboardOutcomes` — body `{ profileIds:[...] }`; classifies conversation outcomes from Supabase `n8n_chat_histories` into the `conversation_outcomes` table (5 statuses: `order_collected`/`owner_needed`/`in_dialog`/`client_silent`/`question_closed`) and returns `{ success, classified, truncated, outcomes:[{profileId,chatId,outcome,summary,outcomeAt,lastMessageAt}] }` (timestamps are unix-ms numbers). Backs the «Сводка» dashboard (`DashboardPage`); on-demand only — no change to the hot bot-reply path. WhatsApp-only in v1

## Additional Instructions

- All API calls use `UnityWebRequest` with coroutines (`IEnumerator` + `yield return SendWebRequest()`). Follow this pattern for any new network calls.
- Never hardcode API keys or tokens directly in source files. All secrets are loaded from `Assets/StreamingAssets/secrets.json` via the `Secrets` class (see `Secrets.cs`). Use `secrets.json.example` as a template.
- When adding new API endpoints, follow the existing pattern: build URL with string interpolation, set auth header, send request, check `request.result`, parse JSON response with `JsonConvert.DeserializeObject`.
- Chat data flows through serializable model classes (`ChatDialog`, `NormalizedMessage`, `RawMessage`) — keep API response models in `Assets/Scripts/Chat/`.

## Conventions

- Singleton pattern for managers (`Manager.Instance`, `ChatManager.Instance`, `BotsPage.Instance`)
- MonoBehaviour with `[SerializeField]` for UI references
- PascalCase for classes, methods, and properties
- Prefab-based reusable components in `Assets/Prefabs/`
- Platform-specific code isolated in bridge classes (`AndroidBridge`, `IOSBridge`)
- TMPro for all text rendering
- DOTween for UI animations
- Bot entity persistence: `PlayerPrefs` keyed by `transform.name` (e.g. `Bot0Name`, `Bot0Products0`)
- Event-driven UI: views subscribe to `ChatManager` events rather than polling
- Partial classes for large components: see `BotSettings.cs` / `BotSettings.Auth.cs`
