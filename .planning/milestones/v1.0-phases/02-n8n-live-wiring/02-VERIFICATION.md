### Confirmation-Bias Counter-Check (deliberate disconfirmation pass)

1. **Partially-efficient requirement:** N8N-04's "hardened against malformed/injected output" is functionally complete but not cost-hardened — WR-03 means a garbage/malicious `POST` to the unauthenticated `/webhook/SuggestReplies` still burns 1-2 real gpt-4o-mini calls before landing on the correct `generation_failed` response. Correctness holds; cost/DoS-amplification resistance does not. Worth a follow-up, not a phase-goal blocker.
2. **Untested error path:** `N8nSuggestionsProvider.Run`'s post-drain re-resolution/chat-mismatch guard (the exact code WR-01 flags) has **zero automated test coverage** — it's a `MonoBehaviour`/coroutine-dependent path that cannot run in EditMode (the plan's own SUMMARY acknowledges `TryGetRecentMessages` "is a MonoBehaviour-private accessor not reachable from EditMode"). This is exactly the kind of path where a real bug (WR-01) hid until manual code review caught it — automated tests alone would not have found it.
3. **Test that passes but is intentionally lenient:** `Steer_Null_SerializesAsJsonNull` accepts EITHER JSON `null` or an absent key as valid — this is a deliberate design choice (matches the plan's own "null → JSON null (or absent)" spec), not a false-positive test, but it's worth flagging that this test cannot distinguish a regression that silently dropped the field entirely from correct behavior.

None of these three findings change the overall verdict — they're documented for honesty per the verification discipline, and none contradicts the roadmap Success Criteria evidence above.
