# Phase 1 ŌĆö UI Design Contract

> Visual and interaction contract for the Reply Suggestions Panel (semi-auto mode), built on a `MockSuggestionsProvider`. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Platform note:** This is a Unity 6 mobile app, single canvas `Assets/Scenes/Main.unity`, `CanvasScaler = Scale With Screen Size`, reference resolution **1080├Ś1920**, `Match = Width (0)`. **All sizes below are in canvas reference units (Ōēł dp ├Ś 3), NOT CSS pixels.** This is the load-bearing rule from `unity-ui-builder` ŌĆö values copied from web mockups render ~Ōģō too small. The 8-point spacing scale in this doc is therefore expressed on the project's measured **4dp grid ├Ś 3 = 12-unit grid**.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Unity canvas UI ŌĆö no shadcn/web design system) |
| Preset | not applicable |

## Typography

Measured TMP scale from `unity-ui-builder` (reference units, NOT px). Exactly 4 roles, 2 weights.

| Role | Size | Weight | Line Height | Used for |
|------|------|--------|-------------|----------|
| Card reply text (body) | 42 | Regular (400) | 1.2 (TMP default leading; ~1.2) | The suggestion reply text ŌĆö **capped at ~2 lines, ellipsis truncation** (D-05 / PANEL-06) |
| Intent chip label | 28 | Semibold (600) | 1.0 | The intent label inside the chip (e.g. ┬½ą”ąĄąĮą░┬╗, ┬½ąØą░ą╗ąĖčćąĖąĄ┬╗, ┬½ą¤čĆąĖą▓ąĄčéčüčéą▓ąĖąĄ┬╗) |

**Weights:** Regular (400) + Semibold (600). No other weights. **Hierarchy comes from size + weight + opacity**, never from adding weights.

## Color

The panel must read as a **calm, native part of the existing WhatsApp chat surface** ŌĆö it is not a new brand zone. Colors below are taken from the project's existing constants so the panel blends in. The 60/30/10 split is measured against the panel itself.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) ŌĆö panel surface | `#FFFFFF` (white) | The suggestions panel background sheet ŌĆö a clean light surface sitting on the warm doodle wallpaper, matching the composer/input surface. Null-sprite Image + RoundedCorners (top corners only feel) |
| Secondary (30%) ŌĆö card surface | `#F5F2EA` (paper, the LOCKED doodle paper tone) ŌåÆ render as a near-white warm card `#FAF8F3` if more contrast vs panel is needed | The 4 suggestion cards: a subtle warm off-white card on the white panel so cards are distinguishable without heavy borders. Use a 1u hairline `#E5DAC6` (locked doodle ink tan) if separation reads weak |
| Accent (10%) ŌĆö semi-auto + steering accent | `#25D366` (WhatsappBrandColor, the project's existing WhatsApp green `Color32(0x25,0xD3,0x66)`) | RESERVED for exactly: (1) the semi-auto top-bar toggle in its **lit "on"** state (D-09 ŌĆö the lit toggle IS the mode indicator), and (2) the "Recommended" badge fill on the top card. Nothing else uses green |
| Tertiary ŌĆö intent chip | `#ECEFF1` fill (muted neutral grey-blue) + `#54656F` label text (WhatsApp secondary-text grey) | The single-accent intent chip (D-06): ONE consistent muted tone for ALL intents ŌĆö **never per-category colors**. The chip is deliberately NOT the green accent (keeps green meaningful for on-state + recommended) |
| Body text | `#111B21` (near-black, matches chat text `0.13,0.13,0.13`) | Card reply text |
| Secondary text | `#54656F` (WhatsApp grey, ~70% neutral) | Chip label, empty/error secondary copy, the off (unlit) toggle icon |
| Destructive | not applicable | No destructive actions in this phase (see Copywriting) |

**Accent reserved for (explicit list ŌĆö never "all interactive elements"):**
1. Semi-auto top-bar toggle, **lit/on state only** (off state is neutral grey `#54656F`).
2. "Recommended" badge on the top card only.

**Notes:**
- The badge and the lit toggle are the ONLY two green elements. Tapping a card does NOT turn it green ŌĆö selection is expressed by the composer filling + a fresh re-clustered set arriving (D-01), not by a persistent selected-card color.
- Skeleton shimmer uses neutral greys only (`#ECECEC` base ŌåÆ `#F5F5F5` highlight sweep), no accent.
- Match shadow/elevation to the warm background; if any card elevation is used, keep it a very soft tinted shadow, never pure black (project memory + mobile-app-ui-design rule). Prefer the hairline over a drop shadow given the doodle wallpaper.

## Interaction & State Contract

| Trigger | Behavior | Requirement |
|---------|----------|-------------|
| Flip top-bar toggle ON | Panel slides in (DOTween), shows 4 skeleton cards ŌåÆ first suggestion set; toggle lit green | SEMI-01, SEMI-03, D-08/09/10 |
| Flip top-bar toggle OFF | Panel slides out / hidden; composer remains; no confirmation; no draft loss | PANEL-05 (reinterpreted, D-11) |
| App restart / bot switch | Per-chat semi-auto state restored from PlayerPrefs `{botId}_semiAuto_{chatId}`; only semi-auto chats show the panel | SEMI-02, SEMI-03 |
| Tap a suggestion card | Load card text into composer (**overwrites** current draft ŌĆö deliberate action) AND request a fresh steered set of 4 (re-cluster); skeleton shimmers in place during regen; never auto-sends | INT-01, INT-04, D-01/02/03/12 |
| Incoming customer message (`OnLiveMessagesReceived`) | Always-expanded panel re-populates (skeleton ŌåÆ fresh set); **never overwrites a dirty composer draft** (automatic action defers) | INT-02, D-02/13 |
| Tap manual-refresh control | Request a fresh set; skeleton in place ŌåÆ new 4 | INT-03 |
| Edit + Send | Reuses existing composer Send button (`ChatManager.SendTextMessage`) | INT-01 |
| Stale / out-of-order / superseded response (rapid picks, chat switch) | Discarded via correlation/sequence guard (model on `CrossChatResponseGuard` / `QuoteResolve` serial-guarded pull); never renders crossed or stale cards | DATA-03 |
| Loading (first + every re-cluster) | 4 shimmer skeletons in place; panel never collapses to empty | PANEL-04, D-12 |
| Empty result | Empty state copy inside panel; composer usable | PANEL-04 |
| Error result (mock error path) | Error copy + ┬½ą×ą▒ąĮąŠą▓ąĖčéčī┬╗ retry inline | PANEL-04, D-15 |

**Visual source-of-truth invariant:** nothing in the panel UI references n8n, UnityWebRequest, or Wappi ŌĆö all suggestion data arrives through `ISuggestionsProvider`. The same UI must light up identically against `MockSuggestionsProvider` (Phase 1) and `N8nSuggestionsProvider` (Phase 2) with zero UI edits.
