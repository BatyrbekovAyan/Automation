<objective>
Close **D5** (high): incoming messages never render in the OPEN chat until the owner re-enters it — owner-confirmed on BOTH WhatsApp and Telegram. This also starves the «Вместе» panel: cards don't refresh on an incoming (I.1 #3, I.2 #6) and the suggestions payload carries a stale transcript, so suggestions aren't relevant to the last incoming (H2).

Root cause (confirmed during investigation, re-confirm in Task 1): the open chat's messages are synced exactly ONCE, by `SyncLatestMessages` inside `OpenChatRoutine` at chat-open. There is **no periodic poll** while a chat stays open. The chat LIST re-syncs only on navigation events (`RefreshActiveBotChats` off `SwipeToBack.OnSlideOutComplete` / the «Чаты» tab), never on a timer, so a message that arrives while a chat is open is invisible until the chat is re-opened. Because the suggestions payload is built from `_activeChatCache` (via `TryGetRecentMessages`) — which the one-shot sync populates and then never refreshes — the newest incoming never reaches the payload either.

Purpose: a message that arrives while the chat is open must appear on its own, within one refresh cycle, on both channels — and because `SuggestionsController.HandleLive` already subscribes to `OnLiveMessagesReceived` and `TryGetRecentMessages` reads the live `_activeChatCache`, a correct poll cascades automatically to card refresh + payload freshness (no separate suggestions change needed).

Output: a pure poll-gate seam + a repeating open-chat poll coroutine that reuses the existing `SyncLatestMessages` (which already computes only brand-new messages and fires `OnLiveMessagesReceived`), wired to start/stop on the chat-open lifecycle and respecting every Wappi response-crossing invariant.

OUT OF SCOPE (do NOT build): push-based delivery (n8n → device push) — recorded as a v2 design item in STATE Deferred Items. This plan fixes the existing client refresh path only. No server/n8n changes.
</objective>
