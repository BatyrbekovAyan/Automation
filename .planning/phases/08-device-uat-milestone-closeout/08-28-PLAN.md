Close **D16** (round-4 NEW, promotion of a documented follow-up): a bot that already has WhatsApp shows
NO Telegram post-creation sync cover when Telegram is authorized later (owner 2026-07-20: "if whatsapp
channel exists and telegram channel is created its sunc cover page is not shown"). NOT a regression -- the
08-19 summary explicitly noted "late-channel auth stamps NO window on EITHER channel (exact parity;
follow-up if ever wanted)".

Root cause (08-REVIEW IN-01): `{bot}TelegramSyncUntil` is stamped ONLY in the creation wizard
(`Manager.CreateBotFromForm`, `Manager.cs:1490-1497`, gated on the creation-time `useTelegram`). Adding
Telegram to an existing WhatsApp bot completes through `ShowAuthSuccess`'s settings-reauth branch
(`!isCreatingBot && Manager.openBot != null`, `Manager.cs:1690`), which never stamps a window ->
`ChatManager.IsChannelSyncing` reads "0" -> no cover. `SyncingView` itself is correct (its `OnEnable`
catch-up + `HandleSyncing` already resume any per-channel window) -- the gap is purely the missing stamp.

Fix: stamp `{bot}TelegramSyncUntil` (same 300s `ChatManager.WhatsAppSyncWindowSeconds`) at the late-auth
success point, ONLY when the re-authed channel is Telegram (`authPage == TelegramAuth`). Reuses the
already-tested key contract (`SyncUntilSuffixFor(Telegram) == "TelegramSyncUntil"`, pinned by 08-19
`ChannelSyncGateTests`; `Bot.DeleteBot` already clears the key). The cover then fires via the existing
fire gates + `SyncingView.OnEnable` catch-up when the user opens the newly-authed Telegram channel.

**Parity decision (documented, deliberate -- per IN-01):** stamp ONLY on late TELEGRAM auth, NOT on late
WhatsApp auth. Stamping WhatsApp here would newly surface a WhatsApp cover where none has ever shown (an
UNREQUESTED behaviour change that breaks the WhatsApp byte-identical invariant this milestone holds).
08-19 kept exact no-stamp parity on both channels; D16 deliberately breaks it in Telegram's favour ONLY,
because the owner reported the missing Telegram cover and the WhatsApp half is neither reported nor wanted.
If the owner later wants the WhatsApp late-auth cover too, that is a separate opt-in change.

Purpose: the Telegram cover fires on late Telegram auth (D16) while WhatsApp stays byte-identical, so
Gate A can close. Output: a one-branch stamp in `ShowAuthSuccess`.
