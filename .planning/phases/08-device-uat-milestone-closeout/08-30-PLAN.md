REFUTED the view layer (08-22/08-26 exonerated) and RELOCATED the defect UPSTREAM to
`TelegramReactionMerge`: the captured `[D2-view]` logs show a healthy post-render on the FIRST remote
change, then NO change event at all for the two subsequent own-reaction changes — echo-without-event.

Root mechanism (08-REVIEW CR-01, CONFIRMED, one code path): the optimistic grace in `Merge` is keyed on
identity + tap-time only and is **never ended by server confirmation**. `result[serverMine] = mine`
(TelegramReactionMerge.cs:62) carries the same fresh `mine` object forward on every ~3 s poll, so even
AFTER the server has echoed the optimistic emoji (grace's job done) any genuinely NEW own-reaction change
made in the Telegram app is still discarded as a "stale echo" for the rest of the 90 s window → the erased
diff makes `SameReactions` true → no `OnMessageReactionsChanged`.

Fix (per CR-01, one 40-line function): **confirmation-clears-grace** — when the server echo carries the
SAME emoji the owner set, ADOPT the server entry (time=0, non-fresh, still "me") instead of replacing it
with `mine`; keep replacing ONLY when the echo DIFFERS (that is the original stale-old-emoji echo the grace
exists to suppress — clearing on differ would regress the D2 flicker). Apply the same confirmation rule to
the fold branch. Also address **WR-01** (same file, the tombstone mirror): DROP the removal tombstone once
the server confirms the absence, instead of carrying it for the full 90 s and suppressing an external own
re-add.

Purpose: an own-reaction change made in the Telegram app repaints the bubble (or the residual is
discriminable via the new `[D2-merge]` log). Output: the confirmation-clears-grace merge + WR-01 tombstone
drop + tests. WhatsApp byte-identical (this is all Telegram merge code — WhatsApp never reaches `Merge`).
