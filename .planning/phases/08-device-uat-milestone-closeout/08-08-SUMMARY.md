**Lifecycle-null-safe SwipeToDelete (lazy `Rt`/`_scroll`) kills the post-bot-creation NRE, and a pure `ChatRowSwipePolicy` gate removes the swipe affordance on Telegram rows — WhatsApp byte-identical, suite 1065/1065 green FRESH.**

## Accomplishments

- **D6 (NRE) fixed** — `SwipeToDelete` is now safe when `ChatItemView.Bind → ResetClosed → SetContentX` runs before `Awake` (the row is `Instantiate()`d while the chat-list panel is inactive, so `Awake` — and the `_rt` assignment — is deferred). A lazy `Rt` accessor (`_rt != null ? _rt : (_rt = (RectTransform)transform)`) now backs every `_rt` dereference; `_scroll` is resolved lazily (`_scroll ??= GetComponentInParent<ScrollRect>()`) at both drag entry points. `Awake` still assigns on the normal path. Channel-agnostic — the crash fired on the WhatsApp path too.
- **D4 (TG swipe affordance) removed** — new pure `ChatRowSwipePolicy.Enabled(channel)` (WhatsApp-only; mirrors `ActiveChannelSupportsChatDelete`). `ChatItemView.Bind` reads `ChatManager.Instance.ActiveChannel` and, on Telegram, snaps the row shut, disables the `SwipeToDelete` component (so no drag callbacks fire → no slide) and hides the red delete button. On WhatsApp it keeps `ResetClosed()` + wires the delete button exactly as before.
- **Pooled-row safety** — the enabled state is set explicitly on BOTH branches every `Bind`, so a row that went to Telegram (swipe off, button hidden) reverts fully when it rebinds as a WhatsApp row.
- **Full EditMode suite green FRESH** — 1065/1065 (baseline 1063 + 2 new `ChatRowSwipePolicy` tests), 0 failed, via the in-Editor bridge (`editorAssemblyWrittenUtc` 2026-07-16T12:04:47Z postdates the last edit at 12:02:39Z).

## D4 channel-gate wiring (in `ChatItemView.Bind`)

- Reads `ChatChannel rowChannel = ChatManager.Instance != null ? ChatManager.Instance.ActiveChannel : ChatChannel.WhatsApp;` (null-guarded → default WhatsApp for Editor/tests).
- `bool swipeEnabled = ChatRowSwipePolicy.Enabled(rowChannel);`
- `swipeToDelete.ResetClosed(); swipeToDelete.enabled = swipeEnabled;` (snap shut first, then gate).
- `deleteButton`: on WhatsApp `SetActive(true)` + wire `OnDeleteClicked` (unchanged); on Telegram `SetActive(false)` (hidden).

## Decisions Made

- **D6 as a lifecycle guard, not a pure seam** — per the plan, the null-init is MonoBehaviour lifecycle (no side-effect-free unit to extract); verified by grep acceptance + full-suite regression rather than a bespoke test.
- **D4 mirrors the existing network policy** — `ChatRowSwipePolicy.Enabled == (channel == WhatsApp)` matches `ActiveChannelSupportsChatDelete`, so the visual affordance and the network guard can never disagree.
- **WhatsApp byte-identical via idempotent no-ops** — the only additions on the WhatsApp path (`swipeToDelete.enabled = true`, `deleteButton.gameObject.SetActive(true)`) are no-ops on a row that has only ever been WhatsApp; they exist solely to restore a pooled row that was previously bound to Telegram.
