Close **D12-ext** (round-4 residual of D12): the create-first-bot CTA works initially (08-18 fix)
but after a WhatsApp↔Telegram chip switch it goes dead on BOTH channels (owner: "works, but stops
working when whatsapp/telegram chip is switched, both whatsapp and telegram create first bot button
stops working."); it heals only on tab re-entry.

Root mechanism (08-REVIEW CR-01, high confidence): on a channel-chip switch with zero bots,
`SetActiveChannel` fires `OnActiveChannelChanged` (→ 08-18's `HandleActiveChannelChanged` re-wires
the CTA correctly), then — in the SAME synchronous stack — calls `BeginLoadForActiveBot`, which with
zero bots resolves `FindBotByName("_default") == null` and fires the **wrong** reason:
`NoConnectionEmptyState()` = `BotHasNoWhatsApp`/`BotHasNoTelegram` instead of `NoBotsExist`
(`ChatManager.BotState.cs:296-300`). That DIFFERENT reason passes the `_lastReason` guard, so
`HandleEmptyState` re-configures the card to «Подключить …» and RE-WIRES the button to
`OpenCurrentBotAuth` — which early-returns silently with no bot (`EmptyStateView.cs:305-307`). The
CTA is inert on both channels, and 08-18's `[D12]` ENTRY log never fires because the wired handler is
no longer `OpenCreateBotFlow`.

Fix (per CR-01, single-view defense — does NOT touch the out-of-scope `BeginLoadForActiveBot`):
coerce the reason via the authoritative `ComputeCurrentEmptyState()` so **NoBots wins**, and log the
silent early-returns. Fold in **08-REVIEW WR-02** (same neighborhood, same coercion path): make
`HandleActiveChannelChanged` re-DERIVE the reason from `ComputeCurrentEmptyState()` (Hide when the new
channel has no empty state) instead of replaying stale `_lastReason` — this kills the wrong-channel
card that can otherwise sit (raycast-blocking) over the Telegram syncing cover for minutes.

Purpose: the primary onboarding CTA is reliable across channel switches so Gate A can go PASS.
Output: a pure NoBots-coercion seam + its wiring + the WR-02 re-derive + non-silent CTA warnings.
