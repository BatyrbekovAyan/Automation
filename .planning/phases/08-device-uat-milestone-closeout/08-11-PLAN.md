<objective>
Close **D2 (medium, RE-FAIL refined)** — Telegram reaction-identity defects that survived 08-06's tombstone fix. On device the owner sees: (1) their OWN single reaction shows count «2»; (2) changing a reaction leaves BOTH the old and new pill; (3) adding ❤️ next to an existing heart renders TWO different heart glyphs.

**Root cause — CONFIRMED from the read-only tapi capture `Tools/tapi/samples/` (two independent bugs converge):**

- **A — Emoji-FORM mismatch.** `Tools/tapi/samples/messages_5258918241.json` shows tapi echoes a heart reaction as the **base form `❤` (U+2764, NO variation selector)**, while the app's quick-bar/catalog stores the **fully-qualified `❤️` (U+2764 U+FE0F)**. Every raw-string emoji comparison then misses: `ReactionSummary.Build` dedups by `emojis.Contains(r.emoji)` so `❤️` and `❤` both render (symptom 3); `TelegramReactionMerge.Key`/`SameReactions` treat them as different; `OutgoingReaction.Resolve` toggle-off (`current == tappedEmoji`) and `ReactionBarController.RefreshHighlight` (`quick[i] == mine`) both miss (08-REVIEW IN-01). Also IN-06: `Key` concatenates `reactorKey+emoji` with no separator.
- **B — Owner-IDENTITY timing.** `TelegramReactionMapper` maps the owner's echoed reaction to `"me"` ONLY when `_tgOwnUserId` is learned (lazily, from a `fromMe` row's `from`) AND `user_id == _tgOwnUserId`. The capture confirms the ids DO match (`from`/`user_id` are both the bare numeric `1038376805`), so the mapping is correct WHEN learned — but `_tgOwnUserId` is reset to null on every bot/channel switch and re-learned only after an own outgoing row is Normalized. React in a chat/page with no own message loaded → `_tgOwnUserId` is null → the owner's echo is keyed by the numeric `user_id`, NOT `"me"` → it survives ALONGSIDE the optimistic `"me"` (`IndexOfMine(server) == -1` → `result.Add(mine)`) → count «2» (symptom 1) and, on a change, both the old numeric-keyed echo and the new `"me"` (symptom 2).

Purpose: an own reaction reads as one reaction, a change replaces cleanly, a removal stays removed — on Telegram, WhatsApp byte-identical.

Constraint: WhatsApp reaction UX byte-identical (WhatsApp reactions never carry a numeric-keyed own echo and its catalog form already matches — normalize for COMPARISON so WhatsApp stored/displayed strings are unchanged). No server/n8n changes. Device confirmation of the live echo bytes + the three symptoms gone rides the 08-16 checkpoint.
