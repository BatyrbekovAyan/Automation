<objective>
Close **D15** (round-4 NEW, pre-existing): a reaction REMOVED in the WhatsApp app itself is never removed
in-app (owner 2026-07-20: "if in whatsapp itself reaction is removed it is still not removed in our app").
WhatsApp add/change already repaints; only removal-propagation is missing. This is PRE-EXISTING (the
WhatsApp `ReactionStore` was deliberately untouched all of v1.1; Telegram removal semantics were built in
08-17) -- NOT a round-4 regression.

Diagnosis (08-REVIEW IN-02, traced end-to-end): the removal MACHINERY is correct -- `ReactionParser`
maps an empty body to `IsRemoval`, and `ReactionStore.ApplyToMessage` removes by `reactorKey` (both
already unit-tested: `FromRaw_EmptyBody_IsRemoval`, `Apply_RemovalDeletesReactorsEntry`). The failure is
at INGEST: WhatsApp live reactions enter ONLY through the UNSEEN-id branch of `SyncLatestMessages`
(`ChatManager.cs:663` `if (seenMessageIds.Add(raw.id))` -> `HandleReactionEvent`, then `continue`). The
already-seen fall-through (`ChatManager.cs:725+`) does media/quote refresh and a Telegram-ONLY reaction
refresh -- it drops a WhatsApp reaction raw entirely. Three candidate shapes remain for a WhatsApp removal
(the code alone cannot discriminate):
- **(a)** Wappi re-emits the removal under the SAME (already-seen) reaction id -> dropped at the seen gate.
- **(b)** Wappi emits no removal raw at all in `messages/get` (platform transport limit).
- **(c)** the removal raw arrives without `stanzaId` -> dropped at `ReactionParser.cs:32`.

Fix (per the project's diagnosis-first pattern -- 08-15/08-17; do NOT guess-fix all three blind):
1. **Instrument** every WhatsApp reaction raw (both branches) with a compiled capped `[D15]` log so the
   round-5 device pass identifies which shape a WhatsApp removal takes.
2. **Fix candidate (a)** -- the highest-code-evidence shape, with a provably-safe idempotent change:
   re-run `HandleReactionEvent` on the already-seen branch for WhatsApp reaction raws. `ReactionStore` is
   idempotent (an unchanged reaction returns false -> no event, no repaint), so this fires only on a
   genuine change/removal and cannot storm. Candidates (b)/(c) are NOT code-discriminable -> left for the
   device logs (no blind fix).

Purpose: WhatsApp reaction removals propagate (candidate a) and a FAIL is discriminable (b/c) so Gate A
can close. Output: the `[D15]` instrumentation + the candidate-(a) already-seen re-process + a regression
pin on the idempotency the fix depends on.

HandleReactionEvent (1737) -- returns false / fires nothing when the event is a buffered/idempotent no-op:
```csharp
private bool HandleReactionEvent(RawMessage raw, IReadOnlyList<MessageViewModel> messages, string liveChatId = null)
{
    ReactionEvent ev = ReactionParser.FromRaw(raw);
    if (ev == null) return false;                              // non-reaction / missing stanzaId -> nothing
    MessageViewModel target = _reactions.Apply(ev, messages);
    if (target == null) return false;                          // buffered, OR idempotent no-op -> nothing
    OnMessageReactionsChanged?.Invoke(target);                 // repaints via MessageItemView.HandleReactionsChanged
    if (liveChatId != null) UpdateChatListPreviewForReaction(liveChatId, raw.id, ev, target);
    return true;
}
```

ReactionStore.Apply / ApplyToMessage -- idempotent: re-applying the same emoji returns false (null from
Apply); a removal on an absent reactor returns false (idempotent re-delivery). ReactionParser.FromRaw
returns null for a missing stanzaId (candidate c) or a non-reaction type. These are already covered by
ReactionStoreTests / ReactionParserTests (removal + idempotent-add cases) -- but NOT the
removal-then-removal-re-delivery no-op the candidate-(a) re-process relies on (Task 2 pins it).
