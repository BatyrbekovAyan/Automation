**LEAD confirmed by inspection (Task 1 re-confirms before implementing):** the WhatsApp cover is `SyncingState`, built by `Assets/Editor/SyncingStateBuilder.cs` into `Screen_Whatsapp/ChatsPanel` (ProgressTrack/ProgressFill = its time-based bar), driven at runtime by `Assets/Scripts/UI/SyncingView.cs`. It never fires for a Telegram-created bot because of THREE WhatsApp-only gates:
1. **No Telegram window is stamped.** `Manager.cs:1443` stamps `{botName}WhatsappSyncUntil` only `if (useWhatsapp)`.
2. **The read key is WhatsApp-only.** `IsWhatsAppSyncing` reads `botId + "WhatsappSyncUntil"` (`SyncUntilKeySuffix`, ChatManager.BotState.cs:216).
3. **The fire is WhatsApp-gated.** `BeginLoadForActiveBot` invokes `OnWhatsAppSyncing` only when `ActiveChannel == ChatChannel.WhatsApp` (ChatManager.BotState.cs:275); `ComputeCurrentEmptyState` gates the same way (:248).

**Maximal reuse — no new scene object.** Phase 6 removed Screen_Telegram; the Telegram list renders in the SAME `Screen_Whatsapp/ChatsPanel` via the channel switcher, so the existing `SyncingState` full-stretch overlay ALREADY sits over the shared chat list. The fix is RUNTIME parameterization (stamp + gate + channel-aware copy), NOT a clone. The gap brief explicitly prefers parameterizing the existing component over cloning it.

Constraint: **WhatsApp cover byte-identical** — the WhatsApp stamp/gate/copy path is unchanged; only Telegram gains a parallel window. Device confirmation rides 08-21.
