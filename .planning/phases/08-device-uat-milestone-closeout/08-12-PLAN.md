<objective>
Close **D9 (low, RE-FAIL)** — the Telegram chat-list sync pill (`ChatListSyncIndicator`, shipped in 08-09 with `OnChatListSyncStart`/`OnChatListSyncEnd`) is NEVER visible on device: the TG list appears instantly and no cue shows.

**This is DIAGNOSIS-FIRST in the Editor** (no device needed to diagnose). The pill code and events are correct in isolation; the failure is a runtime timing/visibility interaction. Enumerated hypotheses (Task 1 confirms which fire):
1. **Sync faster than a visible frame / cached paint** — the list paints instantly from cache and `SyncAllChats` `finally` fires `OnChatListSyncEnd` before the pill renders a single frame at alpha 1 (`Show()`→`Hide()` same frame ⇒ never seen).
2. **Start fires before the pill is subscribed AND the sync ends before OnEnable** — `ChatsPanel` is inactive at scene load; the OnEnable catch-up (`IsChatListSyncing`) only shows if the sync is STILL in flight when the panel activates. A sync that started AND ended during activation is missed.
3. **Channel-read timing** — `HandleSyncStart` reads `IsTelegram()` at fire time; a sync fired while `ActiveChannel` is momentarily WhatsApp (initial resolve before the channel settles to Telegram) is ignored, and no later start re-fires. `HandleActiveChannelChanged` currently only HIDES on non-TG — it never SHOWS a sync already in flight when switching TO Telegram.
4. **Z-order / alpha occlusion** — the pill (last child of ChatsPanel) is covered by list content/mask, or a parent CanvasGroup zeroes its alpha.

Purpose: a Telegram list load reads as a sync (WhatsApp-parity cue), even when the sync is fast. Prefer a CODE-ONLY fix (a minimum-visible-duration + a switch-to-Telegram show); escalate to a scene z-order stamp ONLY if Task 1 confirms occlusion.

Constraint: WhatsApp shows no pill (byte-identical — the start signal stays ignored on WhatsApp). Device confirmation rides the 08-16 checkpoint.
