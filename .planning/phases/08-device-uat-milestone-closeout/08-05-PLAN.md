<objective>
Close **D7** (high): Telegram's own SERVICE dialog (login-code / device-confirmation messages; owner clarification: likely service user `777000`) shows up TWICE in the Telegram chat list — one row with the real Telegram-logo avatar and one with the default silhouette (⇒ two distinct chat-id forms resolving to the same dialog) — AND the same dialog also appears on the WhatsApp page (⇒ cross-channel bleed).

Two independent bugs, both diagnosis-first:
1. **Double id-form:** `ChatManager.ParseChatsJson` keys `chatLookup` by the raw `chat.id`. tapi is returning the service dialog under two different id strings (e.g. a bare numeric `777000` vs a suffixed/prefixed form), so two rows spawn for one dialog.
2. **Cross-channel bleed:** per-channel cache roots are `BotCache/{botId}/chats.json` (WhatsApp) vs `BotCache/{botId}/telegram/chats.json` (Telegram, CHAT-11). The TG service dialog reaching the WhatsApp list means either a Telegram dialog got persisted under the WhatsApp cache root, or a row was synthesized/keyed under the wrong `ActiveChannel`.

Purpose: the service dialog is one row on Telegram and never appears on WhatsApp — without regressing any real WhatsApp chat.

Output: a pure `ChatIdFormat.CanonicalKey` seam that collapses the two id forms, applied at the `chatLookup`/`Chats` insert in `ParseChatsJson`; a verified channel-correct cache path; and regression tests proving the dedup + WhatsApp non-regression.

Constraint: WhatsApp rendering byte-identical. No server/n8n changes. Prod untouched.
</objective>
