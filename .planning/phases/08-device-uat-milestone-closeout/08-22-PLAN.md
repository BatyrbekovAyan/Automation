<objective>
Close **D2-view** (round-4 residual of D2-ext): the Telegram reaction DATA layer is provably
correct (owner: "logs actually always shows correct reaction") but the message-bubble VISUAL
sometimes misses the update. Owner repro: change a reaction on bubble A, then start changing a
reaction on bubble B — bubble A's pill may not repaint.

Root mechanism (08-REVIEW WR-01, moderate-high confidence): the emoji tap renders the pill
**synchronously** (`ChatManager.ReactionSend.cs:61` `OnMessageReactionsChanged?.Invoke(target)`)
while the pressed row still carries the reaction bar's temporary `overrideSorting` Canvas that
`Hide()`→`UnliftRow()` **destroys in the same frame** (`ReactionBarController.cs:181`
`Destroy(_liftedCanvas)` — a deferred destroy landing end-of-frame). A TMP pill mesh regenerated
under the dying nested Canvas can come back with pre-update geometry (the graphic is re-registered
to the root canvas but not re-dirtied). Then it **never self-heals**, because every subsequent
reconcile is deliberately swallowed by the `SameReactions` dedup guard
(`ChatManager.cs:1855` `if (TelegramReactionMerge.SameReactions(...)) return false;`) — the VM data
is already correct, so no further `OnMessageReactionsChanged` ever fires for that message.

Fix (VIEW/refresh layer ONLY — do NOT touch `TelegramReactionMerge`/reconcile; the data layer is
proven correct): after the lifted Canvas is actually gone, re-dirty the pressed bubble's reaction
pill so the mesh regenerates on the root canvas. Add a device-compiled capped log in
`HandleReactionsChanged` for the next UAT pass.

Purpose: the reaction UX finally reads correct on-device so Gate A can go PASS.
Output: a one-frame-deferred pill re-render wired from the reaction bar's dismiss, plus a
compiled diagnostic log.

**Standing rule (shared live tree):** parallel sessions (phases 9/10/11) are committing to this
working tree. Stage ONLY this plan's three files when committing — never `git add -A`.
