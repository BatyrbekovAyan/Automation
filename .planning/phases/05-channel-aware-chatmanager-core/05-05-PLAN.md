<objective>
Fix Telegram 2FA authorization (TGAUTH-01). A 2FA-protected Telegram account currently cannot auth: `SendTelegramCode` only recognizes `auth_success` and shows "Authorization Failed" on a `detail:"2fa"` response, and no `auth/2fa` call exists. Add the cloud-password step: when the code (or QR) flow returns `detail:"2fa"`, reuse the existing code-entry panel as a password prompt (RU copy, relaxed validation), submit `POST tapi/sync/auth/2fa {pwd_code}`, and route success through the existing `ShowAuthSuccess` path. Extract the response-`detail` classification into a pure, unit-tested helper (following the codebase's pure-seam precedent) so the branch logic is testable.

This plan is independent of the ChatManager channel seam — it touches only Manager.cs (the auth flow) plus one new pure helper, and follows the existing hardcoded-tapi auth-URL style (auth/phone, auth/code, auth/qr).

Purpose: 2FA-protected Telegram accounts (very common) can authorize in both the code and QR flows.
Output: a pure TelegramAuthResponseParser + Manager.cs 2FA branches + parser tests.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-0505-01 | Information Disclosure | cloud password handling | mitigate | Password sent ONLY to tapi/sync/auth/2fa over HTTPS with the standard Authorization header; never Debug.Log'd, never written to PlayerPrefs/any store; TelegramCodeInput cleared after submit; _telegram2faMode reset on panel close. |
| T-0505-02 | Tampering | wrong-password / unknown detail | mitigate | Response classification is fail-closed — only a detail starting with auth_success authorizes; every other/malformed detail re-prompts (no false success). |
| T-0505-03 | Denial of Service | malformed auth response body | mitigate | ExtractDetail tolerates malformed JSON (returns "" instead of throwing), so a garbage response can't crash the auth coroutine. |
</threat_model>
