# Vertical Main Prompts — Design

**Date:** 2026-07-06
**Status:** Approved (Approach A)

## Goal

Replace the app's 7 legacy business types with the 6 researched KZ Tier-1 verticals and bake each vertical's prompt into the bot workflow as the **main** system prompt, injected automatically by the n8n Create/Edit workflows based on business type. The Промпт field in BotSettings stays what it is today: Additional Instructions only. No existing functionality breaks — old app builds and legacy bots keep working via fallbacks.

## Current architecture (verified)

- Business types: `Assets/Data/BusinessTypes.asset` (`BusinessTypesSO`), entries `{id, displayName, sprite, tileColor}`. 7 entries, English display names. Wizard tiles built by `Manager.PopulateBusinessTypes`; settings uses `BusinessTypeDropdown`.
- App sends `BusinessType` = displayName in Create (from `selectedBusinessId`, Manager.cs ~2559/~2697) and Edit (from dropdown text, WA/TG edit payload sites) webhook calls, plus `Prompt`, `Business` (pre-prefixed `About Business:\n`), `ProductsList`, `ServicesList`.
- n8n Create (WA `XuvOp7TxOImOAmlj`, TG `Uz6HBBUpAiUqVysB`): clones the bot template by literal id, `Set Fields` writes `nodes[5].parameters.options.systemMessage` =
  `Business Type: {display}\n\n` + template base prompt (`.slice(1)` drops the `=` prefix) + `\n\nAdditional Instructions: {Prompt}\n\nAbout Business: {Business}\n\nProducts:\n{ProductsList}\n\nServices:\n{ServicesList}`.
- n8n Edit (WA `3qax5J9u2qsT9Vao`, TG `TwWPW3gIyjZS3foR`): `Set Fields` preserves the head (slice up to `\nAdditional Instructions: `) and rebuilds the tail; `Set Bussiness Type` swaps only the first line. Consequence today: a business-type change does NOT change the prompt body.

## Decisions (clarified with owner)

1. **Placeholders → About Business.** The six approved templates (docs/prompt-templates-kz-tier1.md) are rewritten as full main prompts; every `[плейсхолдер]` becomes an instruction to take that fact only from the `About Business` section, with lead-capture fallback (имя + телефон + «владелец ответит») when the fact is absent. No per-placeholder onboarding UI.
2. **Prompt map lives in a Code node in each of the 4 workflows.** Canonical texts in `Tools/n8n/prompts/`; `Tools/n8n/inject-prompts.py` writes them into the workflow JSONs. No shared sub-workflow (avoids cross-workflow id references breaking on prod import), no app-side prompt payloads (tuning must not require app releases).
3. **Approach A — fully rebuilt head.** With a known `BusinessTypeId`, both Create and Edit derive the entire head (`Business Type:` line + vertical main prompt) from the map; the systemMessage is a pure function of `(BusinessTypeId, Prompt, Business, Products, Services)`. Hand-tuning a single bot's systemMessage in the n8n UI is not a supported flow and is overwritten on next edit.
4. **Russian display names**, stable kebab-case ids. **Sprites reused** from closest existing icons for now (picked by visual inspection during implementation); proper icons later.

## Design

### 1. Unity

**`Assets/Data/BusinessTypes.asset`** — replace the 7 entries with:

| id | displayName | sprite |
|---|---|---|
| `auto_parts` | Автозапчасти | reuse car_service (or closest on inspection) |
| `wholesale` | Оптовый поставщик | closest existing (e.g. real_estate building) |
| `flowers` | Цветочный магазин | keep flowers |
| `kaspi_seller` | Продавец на Kaspi | closest existing |
| `education` | Учебный центр | closest existing |
| `phone_repair` | Ремонт телефонов | closest existing |

Tile colors: keep/redistribute the existing palette. Old ids (`car_service`, `cafe`, `beauty_salon`, `dentist`, `real_estate`, `tour_agency`) disappear; `IndexOf` returns −1 for saved legacy ids and existing `Mathf.Max(0, …)` call sites fall back to entry 0 — acceptable at dev stage.

**`Manager.cs`** — every payload site that sends `BusinessType` additionally sends `BusinessTypeId`:
- Create WA/TG: `selectedBusinessId` (already resolved via `TryGetById`).
- Edit WA/TG: `businessTypes.TryGetByIndex(BusinessTypeDropdown.value, out var e) ? e.id : ""` (dropdown order mirrors the SO — same source `PopulateBusinessTypeDropdown`).
`BusinessType` continues to carry the (now Russian) display name for the `Business Type:` header line.

**Optional, cuttable:** per-vertical hint as the Бизнес field placeholder text («Укажите: часы работы, адрес, условия доставки, Kaspi рассрочку…») in wizard + settings, stored as a new optional string on `BusinessTypesSO.Entry`. Cut if `EditableTextArea` has no placeholder support — do not grow UI for it.

### 2. n8n workflows (4 files in `Tools/n8n/workflows/`)

Add Code node **`Vertical Prompt`** on the main chain of each workflow:

```js
// PROMPTS-BEGIN (generated by Tools/n8n/inject-prompts.py — do not edit by hand)
const PROMPTS = { auto_parts: "…", wholesale: "…", flowers: "…", kaspi_seller: "…", education: "…", phone_repair: "…" };
// PROMPTS-END
const id = $('Unity Webhook').first().json.body.BusinessTypeId ?? "";
return [{ json: { verticalPrompt: PROMPTS[id] ?? "" } }];
```

- **Create WA/TG** — `Set Fields` head expression becomes a ternary:
  `verticalPrompt` non-empty → `Business Type: {display}\n\n{verticalPrompt}` + existing tail;
  empty → current behavior (template base via `.slice(1)`) + existing tail. Node insertion keeps the existing chain order (webhook → … → Set Fields → create/activate); the Code node must sit before `Set Fields`.
- **Edit WA/TG** — same ternary in `Set Fields`:
  `verticalPrompt` non-empty → head fully rebuilt from map + fresh tail from webhook body;
  empty → current preserve-head + rebuild-tail behavior. The `Set Bussiness Type` first-line swap node is kept unchanged (idempotent under the new path; still needed by the fallback path).
- Tail assembly, `nodes[5]` targeting, webhook paths, template-clone ids, activation calls: unchanged.
- Known cosmetic quirk deliberately untouched: app pre-prefixes `Business` with `About Business:\n`, workflows prepend `About Business:` again (doubled header). Fixing it would break old-app/new-workflow symmetry.

### 3. Prompt sources

- `Tools/n8n/prompts/_core.md` — shared rules appended to every vertical prompt: Chat Memory usage, Supabase Vector Store for products/services/prices, concise mobile-friendly answers, never invent, no «чем ещё помочь» filler, About-Business sourcing rule + lead-capture fallback.
- `Tools/n8n/prompts/{auto_parts,wholesale,flowers,kaspi_seller,education,phone_repair}.md` — the approved templates (docs/prompt-templates-kz-tier1.md) adapted: override phrasing («это исключение из правила…») removed — clarifying-question rules stated positively; every `[плейсхолдер]` replaced with About-Business sourcing + fallback; vertical guardrails (Kaspi no-bypass, no photos/payments/booking, availability hedges, «Если ответа нет» lead capture) preserved verbatim in intent.
- Composition: `map[id] = vertical.md + "\n\n" + _core.md` (vertical role/tone first, hard shared rules last).
- `Tools/n8n/inject-prompts.py` — idempotent; JSON-escapes and rewrites the map between `PROMPTS-BEGIN/END` markers in the `Vertical Prompt` node of all 4 workflow JSONs; fails loudly if a marker or node is missing.
- `docs/prompt-templates-kz-tier1.md` gets a header note: templates are now baked in as main prompts; canonical source = `Tools/n8n/prompts/`; the doc remains as the annotated/rationale version.

### 4. Backward compatibility

| Situation | Behavior |
|---|---|
| Old app build → new workflows | No `BusinessTypeId` → `verticalPrompt=""` → exactly today's behavior (Create: generic template prompt; Edit: preserve head + swap line 1) |
| New app → old workflows | Extra form field ignored; nothing breaks |
| Existing bots (legacy ids) | Dropdown falls back to first entry; live workflow keeps its old prompt until the next settings save, which migrates the head to the new scheme |
| Unknown future id | Empty map hit → same fallback path |

### 5. Rollout & verification

Dev-only; prod bagkz stays dormant (bulk copy later, per standing plan).

1. Regenerate the 4 JSONs via `inject-prompts.py`; push to local n8n via n8n-mcp `update_workflow` (avoids UI-import id-strip), publish.
2. e2e via curl mimicking Unity form posts (multipart — text parts arrive as binary parts in n8n; the workflows already read them correctly, do not "fix"):
   - Create with `BusinessTypeId=flowers` → fetch created workflow → head starts `Business Type: Цветочный магазин\n\nТы — консультант цветочного…` and ends with `_core` before `Additional Instructions:`.
   - Edit with type change `auto_parts → education` → whole head swapped, tail rebuilt.
   - Edit without `BusinessTypeId` → head preserved (legacy fallback), line 1 swapped.
   - Create without `BusinessTypeId` → generic template prompt head (current behavior).
   - Delete test workflows after.
3. Unity: recompile clean; existing EditMode suite stays green (no suite covers these payloads — no new tests forced); wizard tiles + dropdown verified visually by owner (GREEN).

## Out of scope

- Proper vertical icons (stand-ins now, real icons later).
- Fixing the doubled `About Business:` header.
- Per-placeholder onboarding forms; Tier-2 verticals; Kazakh-language prompt variants.
- Any prod n8n changes.

## Acceptance criteria

1. Wizard and settings show exactly the 6 Russian business types.
2. A bot created with any of the 6 types gets that vertical's full main prompt as its systemMessage head; Промпт field content still lands only in `Additional Instructions:`.
3. Changing a bot's business type in settings swaps the entire vertical prompt on save.
4. Requests without `BusinessTypeId` (old app) reproduce today's behavior byte-for-byte in head structure.
5. All 4 canonical workflow JSONs, prompt .md files, and the injection script are committed; local n8n runs the updated workflows; e2e checks above pass.
